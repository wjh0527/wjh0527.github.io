SSRF漏洞

一、SSRF漏洞介绍

SSRF（Sever-Side Request Forgery,服务器端请求伪造）是一种由攻击者构造请求，由服务端发起请求的安全漏洞。它会允许攻击者诱使服务器向攻击者构造的任意域发起一个请求，这个攻击者构造的域一般为内部网络，也就是攻击者常常将存在SSRF漏洞的服务器作为一个“跳板”来对内网其他的服务器进行攻击。

一般情况下，SSRF攻击的目标是外网无法访问的内部系统(正因为请求是由服务端发起的，所以服务端能请求到与自身相连而与外网隔离的内部系统)。

二、SSRF漏洞原理

SSRF形成的原因大都是由于服务端提供了从其他服务器应用获取数据的功能，且没有对目标地址做过滤与限制。例如，黑客操作服务端从指定URL地址获取网页文本内容，加载指定地址的图片等，利用的是服务点的请求伪造。SSRF利用存在缺陷的Web应用作为代理攻击远程和本地的服务器。

首先，我们要对目标网站的架构了解，脑子里要有一个架构图。比如：A网站，是一个所有人都可以访问的外网网站，B网站是一个他们内部的OA网站。我们普通用户只可以访问A网站，不能访问B网站。但是我们可以同过A网站做中间人，访问B网站，从而达到攻击B网站需求。

正常用户访问网站的流程是：

输入A网站URL->发送请求->A服务器接受请求（没有过滤），并处理->返回用户响应

【那网站有个请求是www.baidu,com/xxx.php?image=URL】

那么产生SSRF漏洞的环节在哪里呢？安全的网站应接收请求后，检测请求的合法性。

产生的原因：服务器端的验证并没有对其请求获取图片的参数（image=）做出严格的过滤以及限制，导致A网站可以从其他服务器的获取数据

例如：www.baidu.com/xxx.php?image=www.abc.com/1.jpg

如果将www.abc.com/1.jpg换为与该服务器相连的内网服务器地址会有什么效果呢？

如果存在该内网地址就会返回1xx、2xx之类的状态码，不存在就会返回其他的状态码。

SSRF漏洞就是通过篡改获取资源的请求发送给服务器，但是服务器并没有检测这个请求是否合法的，然后服务器以他的身份来访问其他服务器的资源。

ssrf漏洞的主要危害对象不是存在ssrf漏洞的本服务器，它的主要危害对象是内网的其他服务器。

三、SSRF漏洞的危害

l SSRF可以对外网、服务器所在内网、本地进行端口扫描，攻击运行在内网或本地的应用，或者利用File协议读取本地文件

l 让服务端去访问相应的网址

l 让服务端去访问自己所处内网的一些指纹文件来判断是否存在相应的cms

l 可以使用file、dict、gopher、ftp协议进行请求访问相应的文件

l 攻击内网web应用（可以向内部任意主机的任意端口发送精心构造的数据包payload）

l 攻击内网应用程序（利用跨协议通信技术）

l 判断内网主机是否存活：方法是访问看是否有端口开放

l DoS攻击（请求大文件，始终保持连接keep-alive always）

四、ssrf漏洞的常见存在场景及探测

l 社交分享功能:获取超链接的标题等内容进行显示

l 转码服务:通过URL地址把原地址的网页内容调优使其适合手机屏幕浏览(概率较大)

l 在线翻译:给网址翻译对应网页的内容（概率较大)

l 图片加载/下载:例如富文本编辑器中的点击下载图片到本地;通过URL地址加载或下载图片（概率较大)

l 图片/文章收藏功能:主要其会取URL地址中title以及文本的内容作为显示以求一个好的用具体验（概率较大)

l 云服务厂商:它会远程执行一些命令来判断网站是否存活等，所以如果可以捕获相应的信息，就可以进行ssrf测试

l 网站采集，网站抓取的地方:一些网站会针对你输入的url进行一些信息采集工作

l 数据库内置功能:如mongodb的copyDatabase函数

l 邮件系统:比如接收邮件服务器地址

l 设备后台管理：如进行存活测试

l 未公开的api实现以及其他扩展调用URL的功能:可以利用google高级语法加上这些关键字去寻找SSRF漏洞

l 从远程服务器请求资源

l 其他:打包附件或者内容编辑并导出时、PDF生成或导出

看着很多是不是，其实不用记这么多，下面有一些关键字可能会存在ssrf漏洞，大家遇见关键字就直接试一试。

如：Share、wap、url、link、.rc、 source、target、u、 3g、display、sourceURL、imageURL.domain等。直接把原先参数后面的url换成www.baidu.com或类似的其他的地址进行尝试。

5.2 SSRF漏洞常用协议

l file:/// 从文件系统中获取文件内容，如:file:///letc/passwd

l dict://字典服务器协议，访问字典资源，如:dict://ip:6739/info,常常利用这个协议去探测端口

l sftp:// SSH文件传输协议或安全文件传输协议

l ldap:// 轻量级目录访问协议

l tftp:// 简单文件传输协议

l gopher:// 分布式文档传递服务，可使用gopherus生成payload。可以通过gopher协议将GET请求伪装为POST请求(gopher协议是ssrf利用中一个最强大的协议(俗称万能协议)可用于反弹shell)，是互联网上使用的分布型的文件搜集获取网络协议。gopher协议是在HTTP协议出现之前,在internet上常见重用的协议,但是现在已经用的很少了，利用gopher协议，可以攻击：

• MySQL (Port-3306)

• FastCGI (Port-9000)

• Memcached (Port-11211)

• Redis (Port-6379)

• Zabbix (Port-10050)

• SMTP (Port-25)

l http/s:// 可用于存活探测

六、SSRF漏洞的攻击方式

l 可以对服务器所在的内网环境进行内网或本地的端口扫描(SSRF常配合DICT协议探测内网端口开放情况，但不是所有的端口都可以被探测，一般只能探测出一些带TCP回显的端口)、内网存活探测(http/s)、获取一些服务的banner信息等。

l 通过file协议读取本地文件比如:用户的基本信息【fle:///letc/passwd】，主机名和相应的IP地址【file:l///etchosts】，ARP缓存(内网其他主机)【file:///proc/net/arp】等。

l 对内网web应用进行指纹识别，通过访问应用存在的默认文件实现。

l GET型漏洞利用，GET参数就可以实现的攻击如struts2命令执行、Thinkphp、boss的一些漏洞等，还包括sg注入等。

l POST型漏洞利用，可利用gopher协议进行参数构造。

l 利用Redis未授权访问getshell、Weblogic默认SSRF漏洞页面。

l 如果ssrf漏洞存在于云服务器可以进行攻击元数据服务、攻击存储桶、攻击Kubelet API、越权攻击云平台内其他组件或服务

l DOS攻击等

九、SSRF漏洞修复建议

通常有以下5个思路：

l 过滤返回信息，验证远程服务器对请求的响应是比较容易的方法。如果web应用是去获取某一种类型的文件。那么在把返回结果展示给用户之前先验证返回的信息是否符合标准。

l 统一错误信息，避免用户可以根据错误信息来判断远端服务器的端口状态。

l 限制请求的端口为http常用的端口，比如，80,443,8080,8090。

l 黑名单内网ip。避免应用被用来获取获取内网数据，攻击内网。

l 禁用不需要的协议。仅仅允许http和https请求。可以防止类似于file:///、gopher://、ftp:// 等引起的问题。
