sqlmap的使用

蜡笔小王2025-09-152025-09-15

sqlmap 完整使用指南

本文档基于 sqlmap 实战场景整理，包含基础帮助命令、核心选项详解、实战案例、问题解决及注入防范，适用于 sqlmap 初学者及进阶使用者参考。

一、基础帮助命令

通过以下命令可快速获取 sqlmap 帮助信息，定位所需功能。

命令 作用 备注
python sqlmap.py -h 查看基本帮助信息并退出 包含核心选项的简要说明，适合快速查询
python sqlmap.py -hh 查看高级帮助信息并退出 展示所有选项的详细参数及用法，适合深入配置
python sqlmap.py --version 显示 sqlmap 版本号并退出 确认当前工具版本是否支持目标功能
python sqlmap.py -v VERBOSE 控制注入过程的详细级别 级别范围 0-6，默认 1；6 为最详细，可用于调试

二、核心选项详解

sqlmap 选项按功能分类，以下为各模块关键选项的详细说明。

2.1 Target（目标指定）

至少需指定一个选项定义测试目标，是注入测试的前提。

参数 作用 示例
-u URL, --url=URL 指定单个目标 URL http://www.site.com/vuln.php?id=1
-g GOOGLEDORK 将谷歌搜索结果作为目标 URL 列表 需结合谷歌语法（如 inurl:sqli.php?id=）

2.2 Request（请求配置）

自定义 HTTP 请求参数，适配不同场景（如 POST 请求、代理、Cookie 验证等）。

参数 作用 备注
--data=DATA 发送 POST 请求数据 示例：--data "id=1&name=test"，适用于 POST 型注入
--cookie=COOKIE 指定 HTTP Cookie 头 用于需登录的场景（如 DVWA），示例：--cookie "PHPSESSID=a8d127e; security=low"
--random-agent 随机生成 HTTP User-Agent 避免被 WAF（Web 应用防火墙）识别拦截
--proxy=PROXY 通过代理访问目标 示例：--proxy=127.0.0.1:8080（配合 Burp Suite 抓包分析）
--tor 使用 Tor 匿名网络 保护自身 IP，实现隐私浏览与突破网络审查
--check-tor 验证 Tor 是否正常工作 确保流量正确通过 Tor 节点

2.3 Injection（注入配置）

指定测试参数、自定义 payload 或脚本，缩小测试范围。

参数 作用 备注
-p TESTPARAMETER 指定待测试的参数 多参数场景下仅测试目标参数（如 -p uname），减少无效请求
--dbms=DBMS 强制指定后端数据库类型 示例：--dbms=MySQL、--dbms=SQL Server，避免不必要的跨数据库测试

2.4 Detection（检测配置）

自定义注入检测的深度与风险，平衡测试效率与准确性。

参数 作用 关键说明
--level=LEVEL 测试级别（1-5，默认 1） - 级别 1：不测试 HTTP 头字段（如 User-Agent、Cookie）
- 级别 3+：测试 HTTP 头字段（适用于头注入场景，如 Less-18）
--risk=RISK 测试风险（1-3，默认 1） - 风险 1：无垃圾数据产生
- 风险 3：可能产生大量垃圾数据（如 Insert/Update 注入，需谨慎使用）

2.5 Techniques（注入技术）

指定 SQL 注入类型，减少请求次数，提升测试效率。

参数 作用 支持的注入类型（默认 “BEUSTQ”）
--technique=TECH 选择注入技术 - B：布尔盲注
- E：报错注入
- U：Union 联合注入
- S：堆叠注入
- T：时间盲注
- Q：内联注入

2.6 Enumeration（数据库枚举）

获取数据库信息（版本、用户、表结构等），是注入测试的核心目标。

参数 作用 示例
-a, --all 获取所有数据库信息 一键枚举所有内容（不推荐，效率低）
-b, --banner 获取数据库 Banner（版本信息） 快速确认数据库版本（如 MySQL 5.7.36）
--current-user 获取当前数据库用户 示例结果：root@localhost
--current-db 获取当前数据库名 示例结果：sqli_db
--passwords 枚举数据库用户密码哈希 可结合彩虹表破解明文密码
--dbs 枚举所有数据库名 列出目标服务器上的所有库
--tables 枚举指定数据库的表名 需配合 -D 数据库名（如 --tables -D sqli_db）
--columns 枚举指定表的字段名 需配合 -D 数据库名 -T 表名（如 --columns -D sqli_db -T users）
--dump 导出指定表的所有数据 需配合 -D 数据库名 -T 表名（如 --dump -D sqli_db -T users）
--dump-all 导出所有数据库的所有数据 全面但耗时，需谨慎使用
-D DB 指定目标数据库 配合枚举表/字段/数据使用
-T TBL 指定目标表 配合枚举字段/数据使用
-C COL 指定目标字段 仅导出特定字段数据（如 -C username,password）

2.7 Operating system access（操作系统访问）

突破数据库限制，获取后端操作系统的访问权限。

参数 作用 备注
--os-shell 获取交互式操作系统 Shell 需后端支持文件写入（如 PHP 环境），可执行系统命令
--os-pwn 获取 OOB Shell、Meterpreter 或 VNC 高级功能，适用于渗透测试后期横向移动

2.8 General（通用配置）

设置全局测试参数，简化操作流程。

参数 作用 备注
--batch 自动使用默认值，无需手动交互 批量测试场景必备，避免频繁确认
--flush-session 清除当前目标的会话缓存 重新测试时使用（避免缓存导致的漏测）

2.9 Miscellaneous（其他功能）

辅助功能，适配特殊场景或简化操作。

参数 作用 备注
--sqlmap-shell 进入交互式 sqlmap Shell 支持连续执行多个 sqlmap 命令，无需重复输入 python sqlmap.py
--wizard 启动新手向导模式 分步引导配置测试参数，适合初学者

三、实战案例演示

以下案例基于本地测试环境（如 Pikachu、DVWA），需替换为实际目标 URL 与参数。

3.1 GET 型注入测试

场景：URL 含参数（如 ?id=1），需检测是否存在注入。

命令 说明 关键输出
python sqlmap.py -u http://192.168.200.139/sqli/Less-1/ 无效命令 未指定参数（如 ?id=1），无法触发注入检测
python sqlmap.py -u http://192.168.200.139/sqli/Less-1/?id=1 -v 1 基础测试（详细级别 1） 仅显示关键步骤，适合快速验证
python sqlmap.py -u http://192.168.200.139/sqli/Less-1/?id=1 -v 6 详细测试（详细级别 6） 显示所有请求包、Payload 及响应，用于调试

缓存提示：检测结果会缓存至 C:\Users\admin\AppData\Local\sqlmap\output\目标IP，重复测试需加 --flush-session。

3.2 测试时配置代理

场景：需通过 Burp Suite 抓包分析注入过程。

  • 所有请求会通过 127.0.0.1:8080（Burp 默认端口），可在 Burp 中查看请求详情。
  • 执行中会提示确认数据库类型（如 MySQL），--batch 自动选择默认值。

3.3 POST 型注入测试

场景：表单提交（如登录页），需测试 POST 参数。

案例 1：测试 uname 参数（Less-11）

  • 关键输出：[INFO] POST parameter 'uname' appears to be 'MySQL >= 5.0.12 AND time-based blind' injectable（uname 存在时间盲注）。

案例 2：测试 passwd 参数（Less-17）

  • 关键输出：[INFO] POST parameter 'passwd' is 'MySQL >= 5.0 AND error-based' injectable（passwd 存在报错注入）。

3.4 指定测试参数（-p）

场景：多参数（如 uname、passwd、submit），仅测试 uname。

  • 仅检测 uname 参数，跳过 passwd 和 submit，减少测试时间。

场景：请求复杂（含 Cookie、Referer 等），直接复制 Burp 抓包结果到文件。

  1. 将 Burp 抓包的 HTTP 请求保存为 1.txt（与 sqlmap.py 同目录）；
  2. 执行命令：

3.6 带 Cookie 的注入测试

场景：需登录的系统（如 DVWA），未带 Cookie 会跳转登录页。

  • 推荐方案：将含 Cookie 的请求保存为 1.txt，用 -r 1.txt 测试，避免手动输入 Cookie 出错。

3.7 指定检测级别（–level）

场景：HTTP 头注入（如 Less-18 的 User-Agent 注入），默认级别无法检测。

命令 结果 原因
python sqlmap.py -r 2.txt --batch 未检测到注入 默认级别 1，不测试 HTTP 头字段
python sqlmap.py -r 2.txt --level 3 --batch 检测到注入 级别 3+ 会测试 User-Agent、Cookie 等头字段

3.8 Insert/Update/Delete 请求测试注意事项

场景：留言板、用户注册（Insert 请求），测试可能产生垃圾数据。

  • 问题：--risk 级别 3 时，Insert/Update/Delete 注入会生成大量垃圾数据（如重复留言、无效用户）；
  • 建议：优先测试 Select 请求，避免影响业务数据；若必须测试，需提前告知业务方。

3.9 模拟手机 User-Agent（–mobile）

场景：目标仅对手机端开放注入点，需模拟手机请求。

  • 执行后会提示选择手机型号（如 iPhone 8、Samsung Galaxy S7），默认选 1（iPhone 8）；
  • 适用于移动端专属漏洞测试。

3.10 读取后端文件（–file-read）

场景：获取目标服务器本地文件（如 Windows 配置文件）。

  • 读取结果：保存至 C:\Users\admin\AppData\Local\sqlmap\output\192.168.200.139\files\c__windows_win.ini；
  • 前提：数据库用户有文件读取权限（如 MySQL 的 FILE 权限）。

关键交互步骤：

四、特殊问题解决：sleep() 函数禁用

面试题：当时间盲注依赖的 sleep() 函数被禁用时，如何处理？

解决方案：使用其他延时函数替代，适配不同数据库：

  • MySQL：benchmark(100000000, md5('test'))（通过重复计算消耗时间）；
  • PostgreSQL：pg_sleep(5)（PostgreSQL 专属延时函数）；
  • SQL Server：waitfor delay '0:0:5'（延迟 5 秒）。

五、SQL注入防范方案

5.1 注入产生原因

前端传入的数据未经过滤，直接被当作 SQL 语句执行（如 SELECT * FROM users WHERE id=$_GET['id']）。

5.2 防御方法

  1. 输入过滤：在所有执行 SQL 的位置对输入内容过滤（如过滤 '、;、UNION 等关键字）；

    • 缺点：需修改大量代码，易遗漏边缘场景。

  2. SQL 预编译（推荐）：使用参数化查询，将 SQL 语句与数据分离（如 MySQL 的 PREPARE 语句）；

    • 示例（PHP）：
    • 优点：从根本上杜绝注入，无需手动过滤。

  3. 部署 WAF：通过 Web 应用防火墙拦截注入请求；

    • 推荐工具：
      • IIS 服务器：D盾、安全狗；
      • 通用场景：雷池 WAF（基于反向代理，防护效果好）。
